# Requirements Document

## Introduction

This feature migrates all file storage from Supabase Storage to Cloudflare R2, and adds business logo upload capability. Cloudflare R2 provides S3-compatible object storage with zero egress fees, which aligns with the project's existing Cloudflare Workers deployment. The migration covers three storage domains: AI-generated documents, user-uploaded documents (onboarding files, signatures), and a new business logo upload feature available during onboarding and from the profile settings page. All uploads use short-lived presigned URLs generated server-side via the AWS S3 SDK, ensuring credentials never reach the client.

## Glossary

- **R2_Client**: The server-side S3-compatible client library (`@aws-sdk/client-s3`) configured with Cloudflare R2 credentials, used to interact with R2 buckets and generate presigned URLs.
- **Presigned_URL**: A time-limited, cryptographically signed URL that grants temporary permission to perform a single S3 operation (PUT for upload, GET for download) on a specific object without exposing API credentials.
- **Upload_API**: The Next.js API route (`/api/storage/upload`) that authenticates the user, validates the request, and returns a presigned PUT URL for direct browser-to-R2 upload.
- **R2_Bucket**: The Cloudflare R2 storage bucket named `clorefy` that stores all user files organized by user ID and content type.
- **Object_Key**: The unique path within the R2_Bucket that identifies a stored file, following the pattern `{content_type}/{user_id}/{uuid}.{ext}`.
- **Logo_Uploader**: The React component that provides drag-and-drop or click-to-browse logo upload functionality, used in both the onboarding flow and the profile settings page.
- **Storage_Service**: The shared TypeScript module (`lib/r2.ts`) that encapsulates all R2 operations including presigned URL generation, object deletion, and public URL construction.

## Cloudflare R2 Credentials Required

The following credentials are needed from the Cloudflare dashboard to configure R2 access:

1. **Cloudflare Account ID** — Found on the R2 overview page in the Cloudflare dashboard
2. **R2 Access Key ID** — Generated by creating an R2 API token with "Object Read & Write" permission
3. **R2 Secret Access Key** — Provided once when the R2 API token is created (must be saved immediately)
4. **R2 Bucket Name** — The name of the bucket created for this project (e.g., `clorefy`)

These values must be stored as environment variables / Cloudflare Worker secrets:
- `R2_ACCOUNT_ID`
- `R2_ACCESS_KEY_ID`
- `R2_SECRET_ACCESS_KEY`
- `R2_BUCKET_NAME`

## Requirements

### Requirement 1: R2 Client Configuration

**User Story:** As a developer, I want a centralized R2 client module, so that all storage operations use consistent configuration and credentials are managed in one place.

#### Acceptance Criteria

1. THE Storage_Service SHALL initialize the R2_Client using the `@aws-sdk/client-s3` library with the endpoint `https://{R2_ACCOUNT_ID}.r2.cloudflarestorage.com`, region set to `auto`, and credentials loaded from environment variables `R2_ACCESS_KEY_ID` and `R2_SECRET_ACCESS_KEY`.
2. THE Storage_Service SHALL export a function to generate presigned PUT URLs that accepts a content type, file size limit, and desired Object_Key, and returns a URL valid for no more than 300 seconds.
3. THE Storage_Service SHALL export a function to generate presigned GET URLs that accepts an Object_Key and returns a URL valid for no more than 3600 seconds.
4. THE Storage_Service SHALL export a function to delete an object by Object_Key from the R2_Bucket.
5. IF the R2_Client fails to initialize due to missing credentials, THEN THE Storage_Service SHALL throw a descriptive error at startup rather than failing silently on first use.

### Requirement 2: Presigned URL Upload API

**User Story:** As a user, I want to upload files directly to cloud storage from my browser, so that uploads are fast and don't bottleneck through the application server.

#### Acceptance Criteria

1. WHEN an authenticated user sends a POST request to the Upload_API with a file name, file size, and content type, THE Upload_API SHALL return a presigned PUT URL and the corresponding Object_Key.
2. THE Upload_API SHALL validate that the content type is in the allowed list: `image/png`, `image/jpeg`, `image/webp`, `image/gif`, `application/pdf`.
3. THE Upload_API SHALL validate that the requested file size does not exceed 10 MB.
4. THE Upload_API SHALL generate an Object_Key following the pattern `{content_type_prefix}/{user_id}/{uuid}.{ext}` where `content_type_prefix` is one of `logos`, `documents`, `signatures`, or `uploads`.
5. THE Upload_API SHALL restrict the presigned PUT URL to the exact content type specified in the request by including the `ContentType` parameter in the `PutObjectCommand`.
6. IF the user is not authenticated, THEN THE Upload_API SHALL return a 401 status code.
7. IF the content type or file size is invalid, THEN THE Upload_API SHALL return a 400 status code with a descriptive error message.

### Requirement 3: Presigned URL Download API

**User Story:** As a user, I want to securely access my stored files, so that only I can view my uploaded content.

#### Acceptance Criteria

1. WHEN an authenticated user sends a GET request to `/api/storage/url` with an Object_Key, THE system SHALL return a presigned GET URL valid for 1 hour.
2. THE system SHALL verify that the Object_Key belongs to the requesting user by checking that the user ID segment in the Object_Key matches the authenticated user's ID.
3. IF the Object_Key does not belong to the requesting user, THEN THE system SHALL return a 403 status code.
4. IF the user is not authenticated, THEN THE system SHALL return a 401 status code.

### Requirement 4: Migrate Onboarding Document Uploads

**User Story:** As a user going through onboarding, I want my uploaded business documents (PDFs, images) stored on R2, so that the platform uses a single consistent storage backend.

#### Acceptance Criteria

1. WHEN a user uploads a file during onboarding, THE Upload_Screen component SHALL request a presigned PUT URL from the Upload_API and upload the file directly from the browser to R2.
2. THE Upload_Screen component SHALL display upload progress status (pending, uploading, analyzing, complete, failed) as it does currently.
3. WHEN the upload to R2 completes, THE Upload_Screen component SHALL pass the file to the `/api/ai/analyze-file` endpoint for AI extraction, exactly as the current flow works.
4. THE Upload_Screen component SHALL remove all references to Supabase Storage and the `onboarding-uploads` bucket.

### Requirement 5: Migrate Signature Image Storage

**User Story:** As a document signer, I want my signature images stored on R2, so that all file storage is consolidated on one platform.

#### Acceptance Criteria

1. WHEN a signature is submitted via `/api/signatures/sign`, THE system SHALL upload the signature image to R2 under the `signatures/{signature_id}_{timestamp}.png` Object_Key.
2. THE system SHALL store the presigned GET URL or a reference Object_Key in the `signature_image_url` column of the `signatures` table.
3. IF the R2 upload fails, THEN THE system SHALL fall back to storing the base64 data URL in the database, matching the current fallback behavior.

### Requirement 6: Business Logo Upload During Onboarding

**User Story:** As a new user, I want to optionally upload my business logo during onboarding, so that it appears on my generated documents from the start.

#### Acceptance Criteria

1. WHEN the onboarding flow reaches the final confirmation stage, THE system SHALL present a Logo_Uploader component with a "Skip" option.
2. THE Logo_Uploader SHALL accept image files only: PNG, JPEG, WebP, and GIF formats.
3. THE Logo_Uploader SHALL enforce a maximum file size of 5 MB.
4. WHEN a user selects a logo file, THE Logo_Uploader SHALL display a preview of the image before upload.
5. WHEN the user confirms the logo, THE Logo_Uploader SHALL request a presigned PUT URL from the Upload_API with the `logos` content type prefix and upload the file directly to R2.
6. WHEN the upload completes, THE system SHALL store the Object_Key in the `logo_url` column of the `businesses` table for the user.
7. IF the user skips logo upload, THEN THE system SHALL proceed without modifying the `logo_url` column.

### Requirement 7: Business Logo Upload from Profile Settings

**User Story:** As an existing user, I want to add or update my business logo from the profile settings page, so that I can manage my branding at any time.

#### Acceptance Criteria

1. THE profile page SHALL display a Logo_Uploader component within the Business Information section.
2. WHILE a logo Object_Key exists in the `businesses` table for the user, THE profile page SHALL display the current logo by fetching a presigned GET URL.
3. WHEN the user uploads a new logo, THE system SHALL upload the new file to R2 and update the `logo_url` column in the `businesses` table.
4. WHEN a new logo replaces an existing one, THE system SHALL delete the previous logo object from R2 to avoid orphaned files.
5. THE Logo_Uploader SHALL validate file type (PNG, JPEG, WebP, GIF) and file size (maximum 5 MB) before initiating upload.

### Requirement 8: CORS Configuration for R2 Bucket

**User Story:** As a developer, I want the R2 bucket configured with proper CORS rules, so that browser-based uploads via presigned URLs work correctly.

#### Acceptance Criteria

1. THE R2_Bucket SHALL have CORS rules that allow PUT and GET requests from the application's origin domains: `https://clorefy.com` and `http://localhost:3000`.
2. THE R2_Bucket CORS rules SHALL allow the `Content-Type` header in requests.
3. THE R2_Bucket CORS rules SHALL set a maximum preflight cache age of 3600 seconds.

### Requirement 9: Editor Panel Logo from R2

**User Story:** As a user editing a document, I want my business logo loaded from R2 into the document editor, so that my branding appears on generated documents.

#### Acceptance Criteria

1. WHEN the editor panel loads and the user's business profile has a `logo_url` value, THE editor panel SHALL fetch a presigned GET URL for the logo and display it in the logo section.
2. WHEN the user uploads a new logo via the editor panel's logo upload input, THE system SHALL upload the file to R2 via the Upload_API and update the `logo_url` in the `businesses` table.
3. THE editor panel logo upload SHALL validate file type (image only) and file size (maximum 5 MB) before initiating upload.

### Requirement 10: Security and Access Control

**User Story:** As a platform operator, I want all R2 storage access to be secure and scoped to individual users, so that no user can access another user's files.

#### Acceptance Criteria

1. THE R2_Bucket SHALL remain private (not publicly accessible).
2. THE Storage_Service SHALL generate presigned URLs only on the server side; R2 credentials SHALL NOT be included in client-side bundles or environment variables prefixed with `NEXT_PUBLIC_`.
3. THE Upload_API SHALL enforce user authentication before generating any presigned URL.
4. THE system SHALL include the authenticated user's ID in every Object_Key, ensuring files are namespaced per user.
5. THE Download API SHALL verify that the requesting user's ID matches the user ID segment in the requested Object_Key before returning a presigned GET URL.
6. THE presigned PUT URLs SHALL expire after no more than 300 seconds (5 minutes).
7. THE presigned GET URLs SHALL expire after no more than 3600 seconds (1 hour).
8. THE Upload_API SHALL enforce file type validation by including the `ContentType` parameter in the presigned URL signature, causing R2 to reject uploads with mismatched content types.

### Requirement 11: Environment Variable Configuration

**User Story:** As a developer, I want clear documentation of required environment variables, so that I can configure R2 access in development and production environments.

#### Acceptance Criteria

1. THE system SHALL require the following environment variables for R2 access: `R2_ACCOUNT_ID`, `R2_ACCESS_KEY_ID`, `R2_SECRET_ACCESS_KEY`, `R2_BUCKET_NAME`.
2. THE system SHALL load R2 credentials using the existing `getSecret()` function from `lib/secrets.ts`, supporting Cloudflare Worker secrets, process.env, and Supabase Vault fallback.
3. IF any required R2 environment variable is missing, THEN THE Storage_Service SHALL log a clear error message identifying the missing variable.
4. THE `.env` file SHALL include placeholder entries for all R2 environment variables with comments explaining how to obtain each value from the Cloudflare dashboard.
