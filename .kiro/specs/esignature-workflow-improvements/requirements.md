# Requirements Document

## Introduction

This feature delivers six targeted improvements to the existing e-signature workflow in Clorefy. The current system already supports signature creation, signing pages, audit trails, certificate generation, and notifications. However, several UX and legal-evidence gaps remain:

1. Users can send duplicate signature requests from the same chat/edit session without cancelling the existing one first.
2. The My Documents page does not surface signature status (pending, signed, declined, revision requested) or signer details.
3. Notifications for signature events (signed, declined, revision requested) exist in the backend but the notification types need consistent rendering and navigation.
4. The signing page shows only the document type and expiry date — signers cannot preview the actual contract content before signing.
5. My Documents lacks a way to view full signature details (who signed, when, IP address, verification URL) for legal purposes.
6. There is no way to download a complete legal evidence package (signed document + certificate + audit trail) from My Documents.

All improvements build on the existing Next.js 16 / Supabase / Tailwind / shadcn/ui / Mailtrap / Cloudflare R2 stack and the existing signature infrastructure (`app/api/signatures/`, `lib/certificate-generator.ts`, `lib/signature-audit.ts`, `lib/document-fingerprint.ts`).

## Glossary

- **Signature_System**: The existing e-signature subsystem of Clorefy, covering signature creation, signing page, audit trail, certificate generation, and verification.
- **Document_Preview**: The `components/document-preview.tsx` component that renders the live document preview with a toolbar in the chat/edit screen.
- **GetSignatureModal**: The `components/get-signature-modal.tsx` modal dialog used to create a new signing request from the Document_Preview toolbar.
- **Documents_Page**: The `app/documents/page.tsx` page listing all user documents with status badges and expandable panels.
- **Signing_Page**: The public page at `app/sign/[token]/page.tsx` where external signers draw their signature.
- **Notification_System**: The existing `notifications` table and `app/notifications/page.tsx` page for in-app alerts.
- **Document_Owner**: The authenticated Clorefy user who created and owns the document.
- **Signer**: The external recipient who receives a signing link and signs the document.
- **Signature_Session**: A set of one or more signature records associated with a single `document_sessions` row via `session_id`.
- **Evidence_Package**: A merged PDF containing the original document, the signature certificate page, and the full audit trail for legal purposes.
- **Audit_Trail**: The `signature_audit_events` table containing timestamped records of every action in the signing lifecycle.
- **Certificate_Page**: A PDF page generated by `lib/certificate-generator.ts` summarising signers, timestamps, IP addresses, document fingerprint, and verification URL.
- **DocCard**: The `DocCard` component inside `app/documents/page.tsx` that renders each document row with badges and expandable panels.
- **Signature_Details_Panel**: A new expandable panel within DocCard that displays full signature information for legal reference.
- **Cancel_Confirmation_Dialog**: A confirmation dialog shown when the Document_Owner attempts to cancel an active signature request, requiring explicit confirmation before invalidating the signing link.

## Requirements

### Requirement 1: Prevent Duplicate Signature Requests and Cancel Flow

**User Story:** As a Document_Owner, I want to be prevented from sending a second signature request from the same session when one is already active, and I want the ability to cancel an active request, so that I maintain control over the signing process without creating conflicts.

#### Acceptance Criteria

1. WHILE a Signature_Session has at least one signature record with `signed_at IS NULL` and `signer_action IS NULL` (i.e. pending), THE Document_Preview toolbar SHALL replace the "Request Signature" button with a "Cancel Signature Request" button styled in red/destructive colour.
2. WHEN the Document_Owner clicks "Cancel Signature Request", THE Signature_System SHALL display a confirmation dialog with the message "Are you sure you want to cancel the signature request for [Signer_Name]? The signing link will be invalidated." and two buttons: "Keep Request" (secondary) and "Cancel Request" (destructive).
3. WHEN the Document_Owner confirms cancellation, THE Signature_System SHALL call a cancel endpoint that sets the signature record's `signer_action` to `'cancelled'` and records a `signature.cancelled` audit event.
4. AFTER a signature request is cancelled, THE Document_Preview toolbar SHALL re-enable the "Request Signature" button so the Document_Owner can send a new request.
5. WHILE a Signature_Session has at least one signature record with `signer_action = 'declined'`, THE Document_Preview toolbar SHALL re-enable the "Request Signature" button so the Document_Owner can send a new request to a different signer.
6. WHILE a Signature_Session has at least one signature record with `signer_action = 'revision_requested'`, THE Document_Preview toolbar SHALL re-enable the "Request Signature" button so the Document_Owner can resend after making revisions.
7. WHEN all signature records in a Signature_Session have `signed_at IS NOT NULL` (i.e. all signed), THE Document_Preview toolbar SHALL hide the "Request Signature" button entirely and display only the "Signed" badge and "Download Signed PDF" link.
8. WHEN the Document_Preview component mounts or the session changes, THE Signature_System SHALL fetch the current signature status for the session via `GET /api/signatures?sessionId=` and use the response to determine button state.
9. THE GetSignatureModal SHALL NOT be openable when a pending signature request exists.

### Requirement 2: Signature Status Display in My Documents

**User Story:** As a Document_Owner, I want to see the signature status of each document on the My Documents page, so that I can quickly identify which documents are pending, signed, declined, or need revision.

#### Acceptance Criteria

1. THE Documents_Page SHALL fetch signature data for each document session that supports signatures (contract, quotation, proposal) when loading the documents list.
2. WHEN a document session has at least one pending signature (no `signed_at`, no `signer_action`), THE DocCard SHALL display a "Pending Signature" badge in amber colour with the signer name.
3. WHEN all signatures for a document session have `signed_at IS NOT NULL`, THE DocCard SHALL display a "Signed" badge in green colour.
4. WHEN a signature record has `signer_action = 'declined'`, THE DocCard SHALL display a "Declined" badge in red colour with the signer name.
5. WHEN a signature record has `signer_action = 'revision_requested'`, THE DocCard SHALL display a "Revision Requested" badge in amber colour with the signer name.
6. THE signature status badge SHALL display the signer name and the date of the most recent signature event (signed_at, or created_at for pending).
7. THE Documents_Page SHALL display signature badges alongside existing payment and email badges without layout conflicts.

### Requirement 3: Signature Event Notifications

**User Story:** As a Document_Owner, I want to receive in-app notifications when a client signs, declines, or requests revision of my document, so that I can respond promptly.

#### Acceptance Criteria

1. WHEN a signer successfully signs a document, THE Signature_System SHALL create a notification with type `signature_signed`, title "Document Signed", and message "[Signer_Name] signed your [document_type] [reference_number]." for the Document_Owner.
2. WHEN a signer declines to sign a document, THE Signature_System SHALL create a notification with type `signature_declined`, title "Signature Declined", and message "[Signer_Name] declined to sign your [document_type] [reference_number]." for the Document_Owner.
3. WHEN a signer requests revision of a document, THE Signature_System SHALL create a notification with type `signature_revision_requested`, title "Revision Requested", and message "[Signer_Name] requested revisions to your [document_type] [reference_number]." for the Document_Owner.
4. THE Notification_System SHALL display `signature_declined` notifications with a red X icon and red background styling.
5. THE Notification_System SHALL display `signature_revision_requested` notifications with an amber message icon and amber background styling, including the revision reason text in an expandable detail section.
6. WHEN a Document_Owner clicks a signature notification, THE Notification_System SHALL navigate to the document view page at `/view/[session_id]`.
7. THE Notification_System SHALL include `session_id`, `signature_id`, `signer_name`, `document_type`, and `reference_number` in the notification metadata for all signature event notifications.

### Requirement 4: Document Content Preview on Signing Page

**User Story:** As a Signer, I want to see the actual content of the document before signing, so that I know exactly what I am agreeing to.

#### Acceptance Criteria

1. WHEN a signer opens the Signing_Page, THE Signature_System SHALL fetch the document session context via the signature's `session_id` and render a read-only preview of the document content.
2. THE Signing_Page SHALL display the document content in a scrollable, styled section above the signature pad, showing key fields: parties (from/to names), document description or scope, line items or terms (if present), total amount (if applicable), and any notes.
3. THE Signing_Page SHALL render the document preview in a card layout consistent with the existing Signing_Page design, using the same border, background, and typography styles.
4. IF the document session context is empty or unavailable, THEN THE Signing_Page SHALL display a fallback message: "Document details are not available for preview. Please contact the sender if you need more information."
5. THE document preview section SHALL be collapsible with a "View Document Details" / "Hide Document Details" toggle, defaulting to expanded on desktop and collapsed on mobile.
6. THE Signing_Page SHALL continue to display the existing document type, reference number, and expiry date alongside the new content preview.
7. THE Signing_Page SHALL NOT allow the signer to edit any document content — the preview SHALL be strictly read-only.

### Requirement 5: Editable Signing Invitation Email with Clean Design

**User Story:** As a Document_Owner, I want to preview and edit the email message that will be sent to the signer before sending, so that I can personalise the communication, and I want the email to look clean and professional on all devices.

#### Acceptance Criteria

1. WHEN the Document_Owner clicks "Request Signature", THE GetSignatureModal SHALL display a pre-filled email message preview below the signer name and email fields, showing the exact message that will be sent.
2. THE pre-filled message SHALL follow the pattern: "Hi [Signer_Name], [Business_Name] is requesting your electronic signature on [Document_Type] [Reference_Number]. Please review and sign the document using the secure link below."
3. THE Document_Owner SHALL be able to edit the email message text in a textarea field before submitting the signing request.
4. WHEN the signing request is submitted, THE Signature_System SHALL use the edited message as the `personalMessage` field in the signing invitation email.
5. THE signing invitation email SHALL be fully responsive and render cleanly on mobile devices with a maximum width of 600px, minimum font size of 14px for body text, and touch-friendly button sizing (minimum 44px height).
6. THE signing invitation email SHALL use a clean, modern design with: the business logo and name in the header, a clear document reference card, the personal message (if provided) in a styled block, a prominent "Sign Document" call-to-action button, and the expiry date and security notice in the footer.
7. THE signing invitation email SHALL use a short, clean signing URL format without exposing internal token structure in the visible link text.
8. THE GetSignatureModal SHALL display the signing link sharing options (Copy Link, WhatsApp) only after the request is successfully created, not before.

### Requirement 6: Signature Details Panel in My Documents

**User Story:** As a Document_Owner, I want to view full signature details from My Documents, including who signed, when, their IP address, and the verification URL, so that I have quick access to legal evidence.

#### Acceptance Criteria

1. WHEN a document session has one or more signature records, THE DocCard SHALL include an expandable Signature_Details_Panel accessible via a signature icon button in the action row.
2. THE Signature_Details_Panel SHALL display for each signer: full name, email address, party/role, signature status (pending, signed, declined, revision requested), date and time of signing (formatted as "MMM d, yyyy h:mm a UTC"), IP address used for signing, and a clickable verification URL.
3. WHEN a signature has `signer_action = 'declined'` or `signer_action = 'revision_requested'`, THE Signature_Details_Panel SHALL display the signer's reason text if provided.
4. THE Signature_Details_Panel SHALL fetch signature details from `GET /api/signatures?sessionId=` including the fields: `id`, `signer_name`, `signer_email`, `party`, `signed_at`, `ip_address`, `verification_url`, `signer_action`, `signer_reason`, `created_at`.
5. THE `GET /api/signatures?sessionId=` endpoint SHALL return `ip_address` and `verification_url` fields in addition to the currently returned fields, for authenticated Document_Owner requests only.
6. THE Signature_Details_Panel SHALL use the same expandable animation pattern as the existing PaymentPanel and EmailHistoryPanel in DocCard.

---

### Requirement 7: Legal Evidence Package Download

**User Story:** As a Document_Owner, I want to download a complete evidence package from My Documents that includes the signed document, signature certificate, and full audit trail, so that I have everything needed for legal action in payment disputes.

#### Acceptance Criteria

1. WHEN all signatures for a document session are complete, THE DocCard SHALL display a "Download Evidence Package" button alongside the existing "Signed PDF" link.
2. WHEN the Document_Owner clicks "Download Evidence Package", THE Signature_System SHALL generate a merged PDF containing: the original document (generated from session context), the Certificate_Page (from R2 or regenerated on demand), and an Audit_Trail page listing all `signature_audit_events` for the session in chronological order.
3. THE Audit_Trail page SHALL display for each event: event type (human-readable label), timestamp (UTC formatted), actor email (if available), IP address (if available), and relevant metadata.
4. THE Audit_Trail page SHALL be generated using `@react-pdf/renderer` consistent with the existing Certificate_Page generation pattern.
5. THE Evidence_Package download endpoint SHALL be `GET /api/signatures/evidence/[sessionId]` and SHALL require authentication and verify Document_Owner ownership.
6. THE filename of the downloaded Evidence_Package SHALL follow the pattern: `[referenceNumber]_evidence_[YYYY-MM-DD].pdf`.
7. IF the Certificate_Page has not been generated yet, THEN THE Signature_System SHALL generate it on demand before assembling the Evidence_Package.
8. THE Evidence_Package SHALL include a cover page with: document reference number, document type, all signer names and their signing status, document fingerprint (SHA-256 hash), and a statement: "This evidence package was generated by Clorefy and contains the complete signing record for legal reference."
